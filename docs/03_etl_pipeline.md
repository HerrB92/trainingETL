# ETL Pipeline — Medallion Architecture

## What Is an ETL Pipeline?

**E**xtract, **T**ransform, **L**oad. Data engineering's core pattern:

1. **Extract** — Read data from a source system (here: Azure SQL Database)
2. **Transform** — Clean, validate, reshape the data
3. **Load** — Write it to the destination (here: Delta Lake on ADLS Gen2)

In this project we use the **ELT** variant (Extract → Load raw → Transform progressively), which is more modern and allows re-processing historical data without re-extracting from the source.

## The Medallion Architecture

Named after three medals, each layer has a higher quality standard than the last.

```
Source (Azure SQL)
    │
    ▼
┌─────────────────────────────────────────────────────┐
│  Bronze  — "Raw Data Vault"                         │
│  • Exact copy of source data                        │
│  • Never modified after writing                     │
│  • Append-only or overwrite per table               │
│  • Extra metadata: _source_system, _ingested_at     │
└─────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────┐
│  Silver  — "Clean & Trusted"                        │
│  • Types corrected (string → date, etc.)            │
│  • Strings trimmed, emails lowercased               │
│  • Invalid rows quarantined (not deleted)           │
│  • Deduplication via MERGE INTO (upsert)            │
│  • Enrichment (customers joined with addresses)     │
└─────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────┐
│  Gold    — "Business-Ready"                         │
│  • Star schema (fact + dimension tables)            │
│  • Surrogate keys, SCD2 for slowly changing dims    │
│  • Calculated metrics (revenue, cost_estimate)      │
│  • Exposed via SQL Warehouse for BI tools           │
└─────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────┐
│  GenAI   — "Enriched with AI"                       │
│  • Product embeddings (semantic search)             │
│  • LLM-assigned categories                          │
│  • Stored back in dim_product                       │
└─────────────────────────────────────────────────────┘
```

**Why four layers instead of just one?**
Each layer is independently queryable. If a Silver transformation has a bug, you fix it and re-run Silver from Bronze — you don't need to re-extract from the SQL source. If a Gold aggregation is wrong, you fix it and re-run Gold from Silver. This dramatically simplifies debugging and recovery.

## Data Flow Details

### Bronze: `etl/bronze/ingest.py`

```python
# Reads via JDBC:
spark.read.format("jdbc").option("url", jdbc_url).option("dbtable", query).load()

# Adds metadata:
df.withColumn("_source_system", lit("azure_sql_oltp"))
  .withColumn("_ingested_at", lit(now.isoformat()))
  .withColumn("_batch_id", lit(batch_id))

# Writes as Delta:
df.write.format("delta").mode("overwrite").save(bronze_path)
```

**Watermark-based incremental loading:**
After the first full load, we only read rows newer than the last successful ingestion:
```sql
SELECT * FROM oltp.orders WHERE created_at > '2024-06-01T00:00:00'
```
The watermark timestamp is stored as a Delta table property (`TBLPROPERTIES`).

### Silver: `etl/silver/transform_*.py`

**MERGE INTO (upsert):** Delta Lake's most powerful feature for Silver. Instead of overwriting all data, we merge:

```sql
MERGE INTO silver_customers AS target
USING incoming_customers AS source
ON target.customer_id = source.customer_id
WHEN MATCHED THEN UPDATE SET *
WHEN NOT MATCHED THEN INSERT *
```

This is efficient (reads only changed rows) and preserves history (old versions accessible via Delta time-travel).

**Quarantine pattern:** Invalid rows don't get deleted — they go to a quarantine table with a `_quarantine_reason` column. This is important for auditability: you can always see what data was rejected and why, and reprocess it after fixing the source issue.

### Gold: `etl/gold/star_schema.py`

**Surrogate keys:** In the star schema, every dimension table has a surrogate key (`customer_key`, `product_key`, etc.) — an arbitrary integer generated by Spark (`monotonically_increasing_id()`). The original source ID (`customer_id`) is kept as a "natural key". Surrogate keys decouple the warehouse from the source system's ID scheme.

**Fact table grain:** The `fact_sales` table has one row per order line item. This is the most granular level possible. Reports at higher levels (daily totals, customer totals) are produced by SQL aggregations on top of this table.

**SCD Type 2:** See [04_data_modeling.md](04_data_modeling.md) for the detailed explanation.

```bash
pytest tests/ -v
```

### On Databricks (full end-to-end)

```bash
# Via Databricks CLI (after bundle deploy)
databricks bundle run etl_full_pipeline

# Or run individual stages
databricks bundle run etl_bronze_ingest
databricks bundle run etl_silver_transform
databricks bundle run etl_gold_build
databricks bundle run etl_genai_enrich
```

### Full Refresh vs. Incremental

| Mode | When | How |
|------|------|-----|
| Full refresh | First run, data quality issues, schema changes | Bronze: overwrite; Silver: overwrite; Gold: overwrite |
| Incremental | Daily / scheduled runs | Bronze: append (watermark); Silver: MERGE INTO; Gold: MERGE INTO |

## Delta Lake Features Used

**Time Travel:** Every write to a Delta table creates a new version. Query historical data:
```sql
-- See data as it was 1 week ago
SELECT * FROM delta.`abfss://gold@<acct>/fact_sales` TIMESTAMP AS OF '2024-01-01'

-- See the history of a table
DESCRIBE HISTORY delta.`abfss://gold@<acct>/fact_sales`
```

**OPTIMIZE + ZORDER:** After large ingestions, compact small files and sort data for faster queries:
```sql
OPTIMIZE delta.`abfss://silver@<acct>/products` ZORDER BY (product_id)
```

**Schema Evolution:** When a source column is added, `mergeSchema=true` automatically adds it to the Delta table without breaking existing queries.

## Monitoring and Observability

The `etl/utils/logging.py` module emits structured JSON logs that Azure Monitor can ingest:

```json
{"timestamp": "2024-06-01T06:00:01Z", "level": "INFO", "logger": "etl.bronze.ingest",
 "message": "Ingested", "table": "oltp.orders", "rows": 150, "mode": "incremental"}
```

**What to monitor:**
- `rows = 0` for incremental loads → source might have stopped sending data
- `quarantine_count > 0` → data quality issue in source
- Job duration growing over time → consider OPTIMIZE or partitioning
